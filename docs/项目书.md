# 面向未来事件预测的因果图增强 LLM 项目书

版本：v1.2
日期：2026-07-22
状态：项目唯一总纲；实现细节以代码和专项技术文档为准

具体执行流程、样本级输入输出、训练方法、RL 奖励和评测指标见 [Forecast Trace Pipeline](forecast_trace_pipeline.md)。

## 1. 项目一句话定义

给定预测查询、截止时间、截止时间前可见的新闻文档以及上游提供的结构化事件，先构建与查询相关的事件因果图，再让 LLM 在因果图约束下生成结构化的未来事件演化轨迹，最后完成闭集未来事件预测，并验证这种图约束轨迹是否比原始文档、普通事件列表或无轨迹闭集预测更准确、更稳定、更可解释。

本项目最终目标始终是：

> **让 LLM 根据因果图进行未来事件预测。**

事件抽取位于本方法之外。粗图生成、图 refinement 和 `forecast_trace` 都是服务于最终预测的中间模块。图边分类指标和轨迹格式指标用于诊断模块，但论文结论必须落到下游未来事件预测提升。

## 2. 研究问题

### 2.1 核心问题

LLM 阅读多篇历史文档时，容易受到冗余事实、时间顺序混乱和表面语义相似度影响。项目研究以下问题：

1. 给定严格、可追溯的事件节点，能否构建对未来预测有用的有向关系图？
2. 两阶段的 `coarse graph -> refined graph` 是否比一步生成整图更稳定？
3. refined graph 是否能提高 LLM 对未来事件的预测，而不只是提高静态边分类分数？
4. 让 LLM 在最终答案前输出结构化 `forecast_trace`，是否能改善闭集未来事件预测并提供可检查的中间预测路径？
5. 能否用冻结 LLM 的预测效用作为反馈，让粗图、refinement 和 forecast trace 协同优化？
6. 删除或反转关键因果边后，LLM 的轨迹和最终预测是否发生局部且方向合理的变化？

### 2.2 研究假设

主假设 H1：

```text
LLM(query, documents, refined graph, forecast_trace)
```

在 MIRAI 未来事件预测上优于：

```text
LLM(query, documents)
LLM(query, extracted events)
LLM(query, documents, coarse graph)
LLM(query, documents, refined graph without forecast_trace)
```

辅助假设：

- H2：全局图 refinement 能修正独立事件对分类造成的局部冲突。
- H3：以预测效用为目标的图训练优于只优化静态 edge F1。
- H4：掩码事件预测和局部反事实干预可以提供比全文重构更可靠的图效用奖励。
- H5：结构化 `forecast_trace` 能把因果图从 prompt 附件变成预测路径约束，从而提高闭集预测的准确率和可解释性。

## 3. 任务边界

### 3.1 输入

本方法的正式输入为：

```json
{
  "query": "需要预测的主体、关系或事件类型",
  "cutoff_time": "预测截止时间",
  "documents": [
    {
      "doc_id": "news_001",
      "published_at": "截止时间之前",
      "title": "...",
      "text": "..."
    }
  ],
  "events": [
    {
      "event_id": "e1",
      "trigger": "attacked",
      "mention": "forces attacked the village",
      "document_id": "news_001",
      "sentence_index": 4,
      "participants": ["forces", "village"],
      "confidence": 1.0,
      "evidence": "The forces attacked the village before dawn."
    }
  ]
}
```

事件可以来自数据集 gold annotations 或任意冻结的外部事件抽取器，只要转换为 `event-input-v1`。本项目不训练事件抽取器，也不把抽取效果作为方法贡献。

当前本地评测不依赖 API 和外部 RAG。MIRAI 提供查询及关联文档编号，原生 Qwen 预测器读取原始文档与构建后的图。未来可以替换检索器，但检索本身不是本项目的主要创新点。

### 3.2 输出

最终输出不能只是一张图，而应包含结构化未来事件轨迹和 Top-K 未来事件预测：

```json
{
  "forecast_trace": {
    "intermediate_events": [
      {
        "relative_time": "t-3",
        "event": "opposition groups organize protests",
        "supporting_event_ids": ["e2"],
        "supporting_edge_ids": ["r3"],
        "expected_effect": "raises likelihood of government crackdown",
        "confidence": 0.71
      }
    ]
  },
  "predictions": [
    {
      "event": "预测事件的规范化描述",
      "event_code": "数据集要求的事件代码或类别",
      "actors": ["actor_a", "actor_b"],
      "time_window": "预测时间范围",
      "confidence": 0.78,
      "supporting_event_ids": ["e2", "e5"],
      "supporting_edge_ids": ["r3", "r7"]
    }
  ],
  "abstain": false
}
```

预测必须能追溯到 refined graph 中的支持子图。`forecast_trace` 不是不可验证的自然语言思维链，而是可解析、可干预、可消融的结构化中间预测。若证据不足，系统应允许低置信度或 abstain，而不是强制生成确定答案。

### 3.3 因果表述边界

本项目从观察性文本中学习的是 **evidence-grounded predictive causal graph**，即有助于未来预测的因果假设图。除非数据包含干预或严格识别条件，否则不能声称恢复了真实结构因果模型。

论文中应使用“causal graph hypothesis”“forecast-oriented causal graph”或“causal relation graph”，避免把 LLM 的关系判断描述成严格的因果发现。

## 4. 总体流程

```text
query + cutoff time + historical documents
                    |
                    v
     [upstream, outside our method]
       gold or frozen event extractor
                    |
                    v
       event-input-v1 + evidence spans
                    |
                    v
          [1] 稀疏候选事件对构造      <- our method starts here
                    |
                    v
          [2] Qwen LoRA 逐对关系评分
                    |
                    v
               coarse graph
                    |
                    v
          [3] graph-level refinement
             删边 / 补边 / 改类 / 重估
                    |
                    v
               refined graph
                    |
                    v
          [4] frozen LLM future forecast
                    |
                    v
      Top-K future events + evidence subgraph
                    |
                    v
          [5] downstream evaluation / RL reward
```

工程上保留两个不同规模的 Qwen：

- 图构建 Qwen：默认 0.5B + LoRA。它需要反复处理大量事件对，速度和成本更重要。
- 最终预测 Qwen：建议 4B 或更大。它只在整图生成后执行少量复杂推理，推理能力更重要。

不同参数规模的 LoRA adapter 不能混用。0.5B 训练出的 adapter 必须加载到相同的 0.5B base model。

## 5. 方法输入：严格事件表示

### 5.1 节点定义

图中每个节点必须是具体事件提及，不能是随意截取的一整句。最小字段为：

```json
{
  "event_id": "e1",
  "trigger": "attacked",
  "mention": "forces attacked the village",
  "normalized_event": "forces attack village",
  "participants": ["forces", "village"],
  "doc_id": "news_001",
  "sentence_index": 4,
  "event_time": null,
  "confidence": 0.91,
  "evidence": {
    "sentence": "The forces attacked the village before dawn.",
    "char_span": [11, 38]
  }
}
```

显示格式可简写为：

```text
trigger=attacked; mention=forces attacked the village
```

其中 `sentence` 只作为上下文与证据保存，不直接等同于事件节点。

### 5.2 研究边界与两种设置

代码通过 `event-input-v1` 接收事件，并执行 schema 校验、重复 ID 检查、文档引用检查和置信度检查。完整格式见 [事件输入规范](事件输入规范.md)。

正式实验分为：

1. `Gold-event setting`：MAVEN-ERE 直接使用数据集标注好的 event mentions，用于隔离评估图构建能力。
2. `Frozen-extractor setting`：MIRAI 使用固定的外部抽取器离线生成事件 JSONL，用于评估真实上游噪声下的预测效果。

两种设置都不更新抽取器参数。规则抽取和本地 Qwen 抽取代码只作为兼容基线与数据准备工具，不进入主方法优化器，也不作为论文创新点。

所有上游事件必须来自 cutoff 之前的文档。MIRAI 评测以数据集 query 和 documents 为权威，只读取外部文件中的事件节点，防止事件文件替换上下文或夹带未来标签。

## 6. 阶段 1：候选边与粗图生成

### 6.1 为什么不直接让 LLM 一次输出完整图

“整篇文档 -> 完整图”能利用全局上下文，但存在输出不稳定、漏边不可控、难校准、难定位错误和图规模变化时格式易失效的问题。

“事件对 -> 关系”更容易监督、校准和批处理，但独立判断会缺少全局一致性。

本项目采用混合方案：

1. 上游事件、文档和 query 用于候选范围约束。
2. 对稀疏候选事件对进行可监督的关系评分。
3. 将所有事件对预测组装成 coarse graph。
4. 由 graph-level refinement 负责全局一致性。

未来可以加入文档级 LLM 作为候选边 proposer 或 teacher，但不让它直接替代最终图决策。

### 6.2 候选事件对

当前候选构造主要使用：

- 同文档句子距离不超过 `max_sentence_gap` 的事件对。
- 时间方向允许的有向事件对。
- query 相关性、实体重叠和局部启发式分数较高的事件对。
- `max_pairs` 控制每篇文档的计算量。

候选生成必须首先追求高召回率。若真实边未进入候选集，后续粗图和 refinement 都无法恢复它。

因此论文需要单独报告 `candidate recall@K`，不能只报告进入分类器后的准确率。

### 6.3 粗图关系分类

Qwen LoRA 对每个候选事件对读取：

```text
query（可选）
document title/snippet
source event + mention
target event + mention
```

严格输出：

```json
{"relation_type": "causes", "score": 0.82}
```

当前标签空间为：

```text
none, precedes, causes, escalates, mitigates
```

但 MAVEN-ERE 当前实际监督主要只有：

```text
none, precedes, causes
```

`escalates` 和 `mitigates` 在当前 MAVEN 训练统计中为 0。正式论文有两种合法处理：

1. 只报告并训练
one/precedes/causes`，这是当前最稳妥的主实验。
2. 引入确实包含另外两类关系的数据集后再扩展标签空间。

不能在没有监督数据时宣称模型学会了 `escalates/mitigates`。

### 6.4 粗图输出

多个事件对预测合并为：

```json
{
  "events": ["..."],
  "edges": [
    {
      "source_event_id": "e1",
      "target_event_id": "e2",
      "relation_type": "causes",
      "score": 0.82,
      "candidate_source": "coarse"
    }
  ]
}
```

粗图可以漏边、错边和冲突。它是高召回候选结构，不应被直接当成最终因果图。

## 7. 阶段 2：整图 refinement

### 7.1 任务

refinement 输入完整 coarse graph，在图上下文中联合执行：

1. `keep/drop`：保留或删除粗图边。
2. `add`：从 completion candidates 中补充粗图漏边。
3. `retype`：修正关系类型。
4. `strength`：重估边强度。
5. `frontier`：估计最可能延伸到未来的事件节点。

这里的 `completion` 不是 gold 边，也不是默认成立的 `precedes` 边。推理时它由事件集合的补集经过时间、句距和候选上限筛选得到，只表示“允许 refinement 检查的非粗图边”。`coarse_score` 是候选先验，不代表粗图已经保留该边。

### 7.2 当前模型

当前 `TemporalRelationalEdgeRefiner` 是 time-aware relational GNN：

```text
node features [N, 10]
edge features [E, 20]
query features [6]
       |
       v
hidden_dim = 192
4 rounds bidirectional relation-aware message passing
       |
       v
keep head + type head + strength head + frontier head
```

默认约 3.67M 参数。当前容量对结构特征已足够；下一步优先加入冻结的事件文本语义 embedding，而不是单纯增加 hidden dimension。

专项结构说明见 [refinement模型结构说明.md](refinement模型结构说明.md)。

### 7.3 当前训练缺口

当前 refinement 训练主要由 MAVEN gold graph 人工扰动得到 noisy coarse graph，而实际推理输入来自 Qwen coarse model。两者分布不完全一致，这是当前最重要的 train-inference gap。

正式实验应改为：

1. 用 K-fold 或固定 checkpoint 对 MAVEN 训练集生成 out-of-fold coarse predictions。
2. refinement 只读取真实 coarse predictions 和 gold graph，不读取会泄漏标签的信息。
3. completion candidates 在训练和推理使用相同构造规则。
4. 同时保留人工扰动数据作为增强，而不是唯一训练来源。

这是后续代码完善的 P0 任务。

## 8. 阶段 3：基于 refined graph 的结构化未来事件轨迹预测

### 8.1 模型输入

最终预测 LLM 接收：

```text
system instruction
query
cutoff time
selected historical document evidence
strict event nodes
refined directed edges with relation and confidence
frontier nodes
required JSON output schema
```

主实验建议使用原生 Qwen 4B；图构建继续使用 0.5B。这样将计算预算放在真正需要多跳推理的最终预测阶段。

### 8.1.1 结构化输出

最终预测 LLM 必须先输出：

```text
<forecast_trace>
{
  "intermediate_events": [
    {
      "relative_time": "t-3",
      "event": "...",
      "actors": ["..."],
      "supporting_event_ids": ["e12"],
      "supporting_edge_ids": ["r7"],
      "expected_effect": "...",
      "confidence": 0.71
    }
  ]
}
</forecast_trace>
```

然后输出闭集答案：

```text
<final_answer>
{
  "choice_id": "B",
  "event_code": "...",
  "confidence": 0.74
}
</final_answer>
```

`forecast_trace` 没有人工 gold，不作为独立强监督任务。它的价值通过最终预测提升、图支撑一致性、格式稳定性和干预敏感性证明。

### 8.2 对照输入

为了证明图确实有价值，必须在同一预测模型、相同文档和相同 token 预算下比较：

1. `Documents only`：无图基线。
2. `Extracted events only`：只有事件节点。
3. `Coarse graph`：粗图。
4. `Refined graph`：精图但不强制输出轨迹。
5. `Refined graph + forecast_trace`：图约束轨迹，新的主方法。
6. `Shuffled graph + forecast_trace`：随机打乱边方向或端点的负控制。
7. `Gold/oracle graph`：若测试集允许，仅作为性能上界。

只有 `refined graph + forecast_trace` 显著优于 documents only、coarse graph、refined graph without trace 和 shuffled graph with trace，才能支持“因果图约束未来事件轨迹能增强闭集事件预测”的主要结论。

### 8.3 当前本地评测入口

`src/evaluate_local_qwen_pipeline.py` 已打通无 API 流程：

```text
MIRAI query + dataset documents
-> precomputed event-input-v1
-> Qwen LoRA coarse graph
-> refinement
-> native Qwen graph-conditioned forecasting
-> MIRAI target scoring
```

当前入口默认读取预计算事件；显式设置 `--event-source qwen` 时才运行冻结 Qwen 抽取基线。当前还缺少统一的多对照实验、显著性检验和完整测试集结果。

## 9. 阶段 4：面向 forecast trace 的强化学习

### 9.1 为什么不用全文重构相似度

“图 -> 重构原文 -> 语义相似度”容易出现奖励投机。错误图仍可能通过模糊措辞生成与原文语义相近的文本，因此高重构相似度不等于图正确，也不等于有助于未来预测。

全文重构最多作为弱辅助信号，不作为主奖励。

### 9.2 可训练与冻结模块

RL 阶段：

- 可训练：coarse LoRA policy、refinement policy。
- 固定输入：预计算事件文件；上游事件抽取器不进入训练流程。
- 冻结：最终图条件预测 LLM、奖励评估模型。
- 默认不训练“根据图生成 forecast trace 和最终答案”的大模型主干参数；如需调整，只训练小 adapter 或轻量 forecast head。

冻结预测 LLM 可以减少图策略和生成器共同适应造成的奖励漂移，但仍需负控制和人工审查防止 reward hacking。

### 9.2.1 forecast trace 奖励

`forecast_trace` 的主监督来自最终闭集答案，而不是人工标注的中间事件：

```text
R = w_answer       * R_answer
  + w_support      * R_graph_support
  + w_temporal     * R_temporal
  + w_intervention * R_intervention
  + w_format       * R_format
  - w_density      * R_density
```

其中 `R_answer` 权重最高。辅助奖励只约束轨迹是否可解析、是否引用真实图节点和边、是否遵守时间方向、关键边干预后答案概率是否合理变化。不能让辅助奖励压过最终预测正确性。

### 9.3 掩码事件预测奖励

从历史图中掩码关键事件，冻结 LLM 根据可见事件和图预测被掩码事件：

```text
R_mask = log P(gold_event | visible_events, G)
       - log P(gold_event | visible_events, G_empty_or_shuffled)
```

使用相对增益而不是单独的 Top-K 文本重叠，可减少模型仅凭语言常识猜对事件带来的虚高奖励。

掩码任务是图表示学习的辅助训练信号，不替代 MIRAI 未来事件预测这一最终任务。

### 9.4 局部反事实干预奖励

对关键边执行删除或反转，得到 `G_do`，观察冻结 LLM 的预测变化：

- 受该边影响的后代事件预测应发生方向合理的变化。
- 无关非后代事件预测应尽量稳定。
- 删除弱边的影响应小于删除强边。

奖励由局部敏感性、方向一致性和非后代不变性组成，不能只最大化两次生成文本的全局语义距离，否则模型会偏好任意剧烈变化。

### 9.5 总图效用与阶段信用分配

建议定义：

```text
U(G) = alpha * R_mask
     + beta  * R_counterfactual
     + gamma * R_forecast
     - lambda_cycle    * cycle_penalty
     - lambda_temporal * temporal_violation
     - lambda_density  * density_penalty
```

两阶段奖励：

```text
R_coarse = U(G_coarse) + supervised_edge_reward
R_refine = U(G_refined) - U(G_coarse)
```

`R_refine` 使用增量效用，使 refinement 只在确实改善图时获得正奖励，直接增强两阶段关联性。

当前 `src/rl_pipeline_hooks.py` 只定义 trajectory、step 和 reward policy 接口，尚未实现 PPO/REINFORCE 优化器。后续应先实现离线 reward 计算和 action log-prob，再决定具体 RL 算法。

## 10. 数据集分工

### 10.1 MAVEN-ERE

用途：

- 严格事件节点与关系监督。
- coarse pair classifier 训练与测试。
- refinement 静态图指标训练与测试。
- 直接使用数据集 gold event mentions，不训练事件抽取器。

限制：

- 当前关系类型高度不均衡。
- `escalates/mitigates` 无当前监督。
- 静态关系准确不代表未来预测有效。

### 10.2 MIRAI

用途：

- 项目最终 future event forecasting benchmark。
- 比较无图、粗图和 refined graph 对 LLM 预测的影响。
- 后期提供 forecast-oriented reward。
- 使用同一个冻结外部抽取器预计算 `event-input-v1`，抽取结果在所有图方法对照中保持不变。

严格要求按时间切分，只使用 cutoff 前文档。所有缓存、candidate generation 和 prompt 都要进行泄漏审计。

### 10.3 可选扩展数据

只有在主流程稳定后再引入：

- Causal News Corpus：补充新闻因果关系。
- Event StoryLine Corpus：补充事件链和跨句关系。
- MATRES：补充时间关系。
- 其他带强度或干预信息的数据：支持因果强度与反事实实验。

扩展数据必须统一到严格事件 schema，并分别报告域内与跨域结果。

## 11. 训练方案

### 11.1 Phase A：监督式图建模

1. 从 MAVEN-ERE gold event mentions 构造统一事件输入。
2. 训练 0.5B Qwen LoRA coarse relation scorer。
3. 在 MAVEN test 上计算候选召回、边分类、关系宏 F1 和校准指标。
4. 生成 out-of-fold coarse graphs。
5. 用真实 coarse graphs 训练 refinement。
6. 比较 coarse 与 refined 的静态图质量变化。

### 11.2 Phase B：未来事件预测基线

1. 用固定外部抽取器为 MIRAI validation/test 生成预计算事件文件。
2. 固定 Qwen 4B 预测模型和解码参数。
3. 所有对照复用完全相同的预计算事件。
4. 完成六种图输入对照。
5. 在 MIRAI validation 上选阈值，不在 test 上调参。
6. 在 test 上一次性报告最终结果和 bootstrap 置信区间。

### 11.3 Phase C：预测效用优化

1. 先离线实现 `R_mask` 和反事实奖励，检查是否与 edge F1、forecast metric 正相关。
2. 训练 refinement 的增量效用目标。
3. 再尝试 coarse + refinement 联合 RL。
4. 使用 KL regularization 保持策略接近监督模型。
5. 检查边密度膨胀、模式坍缩和奖励投机。

## 12. 评测指标

### 12.1 事件与候选层

- `event-input-v1` schema 有效率与缺失 query 比例。
- Gold-event 与 frozen-extractor 两种设置分开报告，不把抽取指标作为主方法指标。
- candidate edge recall@K。
- 每图候选数、耗时和显存。

### 12.2 粗图层

- edge existence precision / recall / F1。
- relation accuracy、micro-F1、macro-F1、per-class F1。
- causal-only F1 与 temporal-only F1。
- score calibration：ECE、Brier score。
- graph exact match 仅作补充，不作为主指标。

### 12.3 refinement 层

- refined edge F1 及其相对 coarse 的 `Delta F1`。
- add / delete / retype 分项准确率。
- cycle rate、时间方向违反率、边密度。
- frontier node ranking 指标。
- 按图规模、文档跨度和关系类型分桶分析。

### 12.4 最终预测层

- MIRAI 目标代码或事件类别的 Accuracy / Hit@K。
- 多标签任务的 micro-F1、macro-F1。
- 排序任务的 MRR 或 MAP（按实际 gold schema 选择）。
- calibration、abstain coverage-risk。
- refined graph 相对 documents only 和 coarse graph 的绝对与相对提升。
- bootstrap 95% confidence interval 和配对显著性检验。

最终论文主表必须以预测层指标为中心，图指标放在机制分析表中。

## 14. 论文可成立的创新点

单独的 Qwen 事件对分类或普通 GNN refinement 不足以构成强论文。建议把论文贡献集中为以下三点：

1. **Forecast-oriented coarse-to-refined causal graph**
   将可校准的局部事件对评分与全局图修正结合，并以未来事件预测效用而不是单一 edge F1 定义最终图质量。

2. **Stage-wise graph utility credit assignment**
   使用 `U(G_refined) - U(G_coarse)` 对 refinement 分配增量奖励，使粗图 proposer 与 refinement 在同一预测目标下协同，而不是独立训练后简单串联。

3. **Masked and localized counterfactual graph rewards**
   用掩码事件预测增益和局部边干预一致性替代易投机的全文重构奖励，并冻结最终预测 LLM，降低生成器与图策略共同迎合奖励的风险。

辅助贡献：

- 严格事件 mention、证据 span 和支持子图带来可追溯预测。
- 小模型高频构图、大模型低频预测的成本分工。
- coarse/refined/shuffled/oracle 完整负控制实验。

最终能否形成一篇质量尚可的论文，取决于 MIRAI 主结果、显著性、消融和错误分析，而不是模块数量。项目应优先把一个清楚的假设证明完整，不再扩展无关功能。

## 15. 必做消融实验

1. 无图 vs 事件节点 vs coarse graph vs refined graph。
2. Gold events vs 同一 frozen extractor events，衡量上游噪声影响。
3. refined graph vs shuffled edges vs reversed edges。
4. 无 completion vs 有 completion。
5. 独立 pair scorer vs graph-level refinement。
6. 结构特征 refiner vs 加冻结文本 embedding。
7. 人工扰动 coarse 训练 vs 真实 out-of-fold coarse 训练。
8. 0.5B coarse vs 4B coarse 的质量、速度和显存。
9. 无 `R_mask`、无 counterfactual reward、无 forecast reward。
10. 冻结 predictor vs 联合训练 predictor，后者只作风险对照。
11. 不同关系阈值和图密度下的预测稳定性。

## 16. 风险与应对

### 风险 1：图指标提升但未来预测不提升

应对：把 downstream utility 提前到验证流程；使用 frontier、支持子图和预测增量奖励，而不是继续堆高 edge F1。

### 风险 2：候选集漏掉真实边

应对：先优化 candidate recall@K，再优化分类 precision；加入文档级 proposer 只能作为候选补充。

### 风险 3：refinement 学到人工噪声模式

应对：使用 out-of-fold Qwen coarse graphs，统一训练与推理 completion 规则。

### 风险 4：关系类别极端不均衡

应对：主实验收缩为有真实监督的关系；增加 macro-F1、class weighting 和跨数据集补充实验。

### 风险 5：时间泄漏

应对：数据读取、缓存、事件抽取、候选构造和 prompt 五处分别检查 cutoff；保存每条预测所用 doc_id 便于审计。

### 风险 6：RL 奖励投机

应对：相对基线奖励、局部干预、不相关节点不变性、KL 约束、图密度约束和人工抽样检查同时使用。

## 18. 论文组织建议

暂定题目：

> Forecast-Oriented Coarse-to-Refined Causal Graphs for LLM-based Future Event Prediction

章节结构：

1. Introduction：LLM 事件预测中的噪声上下文与结构推理问题。
2. Related Work：事件预测、文本因果图、LLM 图推理、图结构修正、因果奖励。
3. Task Definition：query、cutoff、documents、graph 和 future events。
4. Method：严格事件表示、coarse scorer、graph refinement、forecast-oriented utility。
5. Experiments：MAVEN 中间评测、MIRAI 主评测、负控制和消融。
6. Analysis：反事实敏感性、图错误传播、成本与局限。
7. Conclusion。

论文最关键的一句话应始终能被主实验直接验证：

> 与无图和粗图基线相比，面向预测效用修正的因果图能否稳定提高冻结 LLM 的未来事件预测。
